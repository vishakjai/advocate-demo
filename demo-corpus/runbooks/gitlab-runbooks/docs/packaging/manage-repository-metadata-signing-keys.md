# Managing Repository Metadata Signing Keys

This document covers the generic process for managing repository metadata signing keys. For Pulp-specific implementation details, see [manage Pulp repository metadata signing keys](../pulp/manage-repository-metadata-signing-keys.md).

## Overview

Repository metadata signing keys are used to sign repository metadata, giving users certainty that the metadata was generated by the repository provider. This process involves:

1. Requesting access to the key management system
2. Generating or extending GPG keys
3. Updating the key in the key management system
4. Cleaning up local copies

## Prerequisites

- Member of the appropriate team (e.g., Build team for PULP)
- Access to Okta for authentication
- GPG tools installed locally

## Process

### 1. Request Access

Create an access request issue to obtain read/write access to the signing key:

1. [Create access request issue](https://gitlab.com/gitlab-com/team-member-epics/access-requests/-/issues/new?issuable_template=Individual_Bulk_Access_Request)
1. For _System(s)_, specify _Okta Group Membership_.
1. For _System Name_, specify the appropriate group (e.g., `Okta Group: Team - Distribution - PULP Repository Metadata Signing Key`).
1. For _Justification for this access_, provide context about the key update needed.
1. Follow the instructions in the issue to get your AR approved & actioned by the provisioners.

### 2. Generate or Extend GPG Key

Choose one of the following:

**Generate a new GPG key pair** if you are rotating the key:

- Follow the [GPG key generation process](./generate-gpg-key-pair.md#generate-the-keys) for your system
- Export the private key in the appropriate format

**Extend the expiry date** if the current key is due to expire and you want to keep it:

- Obtain the existing private key from your key management system
- Import it locally
- [Extend the expiration date](./generate-gpg-key-pair.md#extending-key-expiration)
- Export the updated private key

The outcome of this step should be a new or extended private key ready for deployment.

### 3. Update Key in System

Deploy the key to your key management system. For system-specific instructions, see:

- [Pulp](../pulp/manage-repository-metadata-signing-keys.md#3-update-key-in-system)

### 4. Validate Deployment

After deployment, validate that the key has been successfully updated. For system-specific validation steps, see:

- [Pulp](../pulp/manage-repository-metadata-signing-keys.md#4-validate-deployment)

### 5. Clean Up Local Copies

See [purging local copies](./generate-gpg-key-pair.md#purging-local-copies) for detailed instructions on securely removing the private key from your local machine.

### 6. Revoke Access

Once the deployment is complete and validated, create an access change issue to revoke your temporary access:

1. [Create access change issue](https://gitlab.com/gitlab-com/team-member-epics/access-requests/-/issues/new?issuable_template=Access_Change_Request)
1. For _System_, specify _Okta Group Membership_.
1. For _System Name_, specify the appropriate group.
1. For _Justification for this access change/removal_, explain that the temporary access is no longer needed.
1. Assign the issue to the Okta provisioners as no approval is needed for access removal.
